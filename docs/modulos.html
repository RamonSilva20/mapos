Sistema de Modulos

O sistema de modulos do Map-OS permite estender a aplicacao sem modificar o codigo principal. Um modulo e um pacote ZIP autocontido que pode adicionar novas paginas, executar SQL, injetar conteudo em views existentes e registrar configuracoes — tudo gerenciado pelo painel administrativo.

O que um modulo pode fazer

  • Criar novas tabelas no banco de dados
  • Adicionar controllers e models acessiveis via URL normal
  • Injetar blocos HTML em pontos especificos de views existentes
  • Adicionar itens ao menu lateral
  • Registrar entradas na tabela configuracoes
  • Executar logica arbitraria no install e uninstall via PHP

O que um modulo nao pode fazer

  • Modificar arquivos do core da aplicacao
  • Sobrescrever controllers ou models existentes
  • Injetar conteudo em pontos que nao possuem um hook declarado

Arquitetura

Os arquivos do modulo residem permanentemente em modules/<slug>/ na raiz do projeto. Nenhum arquivo e copiado para dentro de application/ durante a instalacao, com excecao de arquivos wrapper finos para controllers e models.

Componentes do sistema

Componente Localizacao Responsabilidade
Module_manager application/libraries/ Upload, instalacao, desinstalacao, toggle de status
MY_Controller application/core/ Carrega modulos ativos, registra view hooks, popula menu
module_hook() application/helpers/ Executa pontos de injecao nas views existentes
Module_dispatch application/controllers/ Proxy que roteia requisicoes para controllers de modulos
Tabela modulos Banco de dados Registro de modulos instalados e seus status
Tabela modulo_menu_items Banco de dados Itens de menu dinamicos por modulo

Ciclo de vida de uma requisicao com modulo ativo

  1. MY_Controller::load_configuration() executa em todo request autenticado
  2. Consulta a tabela modulos por registros com status = 'ativo'
  3. Para cada modulo ativo, chama add_package_path() com o diretorio do modulo
  4. Le o arquivo hooks/module_hooks.php do modulo e registra os view hooks na config do CI
  5. Popula $data['module_menu_items'] com os itens de menu do banco
  6. Quando uma view chama module_hook('nome_do_hook'), o arquivo de hook correspondente e incluido no escopo da funcao com as variaveis da view disponibilizadas via extract()

Roteamento de controllers de modulos

O CI3 resolve controllers exclusivamente em application/controllers/. Para suportar controllers em modules/<slug>/controllers/, o sistema utiliza duas estrategias em conjunto:

  1. Durante a instalacao, o Module_manager cria um arquivo wrapper thin em application/controllers/ que faz require_once do arquivo real no diretorio do modulo.
  2. Como fallback (caso o diretorio nao tenha permissao de escrita), $route['404_override'] = 'module_dispatch' captura a requisicao e despacha para o controller correto via Module_dispatch.

Fluxo de Instalacao

  1. Upload do ZIP

    O usuario acessa Modulos › Instalar Modulo e faz upload de um arquivo .zip. O Module_manager::upload() extrai o ZIP em um diretorio temporario (modules/tmp_xxx/), realiza o unwrap caso o ZIP contenha uma subpasta raiz, valida o module.json e move o conteudo para modules/<slug>/.

  2. Confirmacao

    O usuario e redirecionado para uma pagina de confirmacao que exibe as informacoes do modulo (nome, versao, autor, descricao), os itens de menu que serao criados, as configuracoes que serao inseridas e se ha SQL de migracao a ser executado.

  3. Instalacao

    O Module_manager::install() executa na ordem:

    1. Executa migrations/install.sql query por query
    2. Inclui install.php do modulo (com $CI disponivel)
    3. Cria wrappers em application/controllers/ e application/models/
    4. Insere itens de menu em modulo_menu_items
    5. Insere configuracoes em configuracoes via INSERT IGNORE
    6. Registra o modulo em modulos com status = 'ativo'

  4. Desinstalacao

    O processo reverso: executa migrations/uninstall.sql, inclui uninstall.php, remove os wrappers, remove configuracoes, deleta o registro do banco e remove o diretorio modules/<slug>/.

Estrutura de Diretorios

modules/ meu_modulo/ module.json <-- obrigatorio install.php <-- executa na instalacao uninstall.php <-- executa na desinstalacao controllers/ Meu_modulo.php models/ Meu_modulo_model.php views/ index.php hooks/ meu_hook.php <-- conteudo injetado em views existentes hooks/ module_hooks.php <-- registra os view hooks migrations/ install.sql uninstall.sql assets/ css/ js/
Convencao de nomenclatura

O slug do modulo deve conter apenas letras minusculas, numeros e underscores. O nome do arquivo do controller deve ser ucfirst(slug).php (ex: slug meu_modulo → arquivo Meu_modulo.php → classe Meu_modulo).

module.json

O module.json e o unico arquivo obrigatorio em um modulo. Ele descreve o modulo e declara todos os recursos que serao provisionados durante a instalacao.

json module.json 
{
  "slug":        "meu_modulo",
  "name":        "Meu Modulo",
  "version":     "1.0.0",
  "description": "Descricao breve do modulo.",
  "author":      "Seu Nome",

  "menu": [
    {
      "title":      "Meu Modulo",
      "url":        "meu_modulo",
      "icon":       "bx bx-extension",
      "permission": "vMeuModulo",
      "tooltip":    "Meu Modulo",
      "ordem":      150
    }
  ],

  "configurations": [
    { "key": "meu_modulo_opcao", "value": "padrao" }
  ],

  "view_hooks": [
    {
      "hook": "os_detail_extras",
      "view": "hooks/meu_hook.php"
    }
  ],

  "screenshots": [
    "https://exemplo.com/img/screen1.png",
    "https://exemplo.com/img/screen2.png"
  ],
  "video_url": "https://www.youtube.com/watch?v=VIDEO_ID"
}

SQL Migrations

Os arquivos SQL sao executados statement por statement, separados por ponto-e-virgula. Use IF NOT EXISTS e IF EXISTS para tornar as migracoes idempotentes.

sql migrations/install.sql 
CREATE TABLE IF NOT EXISTS `meu_modulo_dados` (
  `id`           INT          NOT NULL AUTO_INCREMENT,
  `clientes_id`  INT          NOT NULL,
  `conteudo`     TEXT         NULL,
  `criado_em`    TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `atualizado_em` TIMESTAMP   NOT NULL DEFAULT CURRENT_TIMESTAMP
                              ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  INDEX `idx_clientes_id` (`clientes_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_general_ci;
sql migrations/uninstall.sql 
DROP TABLE IF EXISTS `meu_modulo_dados`;
Atencao

O uninstall.sql e executado antes da remocao do modulo. Certifique-se de incluir DROP TABLE IF EXISTS para todas as tabelas criadas pelo install.sql.

Scripts de Instalacao e Desinstalacao

Os arquivos install.php e uninstall.php sao incluidos pelo Module_manager com a variavel $CI disponivel no escopo, alem de $moduleSlug e $moduleManifest. Use-os para logica que nao pode ser expressa em SQL puro.

php install.php 
// $CI, $moduleSlug e $moduleManifest estao disponíveis

// Exemplo: inserir permissao padrao para o grupo Administrador
$grupo = $CI->db
    ->where('nome', 'Administrador')
    ->get('grupos')
    ->row();

if ($grupo) {
    $CI->db->query(
        "INSERT IGNORE INTO `grupos_permissoes`
         (`grupos_id`, `permissao`) VALUES (?, ?)",
        [$grupo->id, 'vMeuModulo']
    );
}
Nota

Se o modulo nao precisar executar nenhuma logica PHP adicional, basta deixar o arquivo com um comentario vazio. O arquivo deve existir mas pode estar em branco.

Controllers

Controllers de modulos seguem exatamente as mesmas convencoes do CI3. Eles devem estender MY_Controller para herdar autenticacao, carregamento de configuracoes e permissoes.

php controllers/Meu_modulo.php 
if (!defined('BASEPATH')) exit('No direct script access allowed');

class Meu_modulo extends MY_Controller
{
    public function __construct()
    {
        parent::__construct();

        if (!$this->permission->checkPermission(
            $this->session->userdata('permissao'),
            'vMeuModulo'
        )) {
            redirect(site_url('dashboard'));
        }
    }

    public function index()
    {
        $this->data['view'] = 'meu_modulo/index';
        return $this->layout();
    }

    // Endpoint AJAX — retorna JSON, sem layout
    public function get(int $id)
    {
        $row = $this->db
            ->where('id', $id)
            ->get('meu_modulo_dados')
            ->row();

        $this->output
            ->set_content_type('application/json')
            ->set_output(json_encode([
                'dados' => $row ? $row->conteudo : ''
            ]));
    }
}
Convencao de nomenclatura

O nome da classe deve ser ucfirst do slug. Para o slug meu_modulo, a classe e Meu_modulo (nao MeuModulo). O arquivo deve se chamar Meu_modulo.php.

O controller e acessivel via /meu_modulo/index ou /meu_modulo/get/1 sem nenhuma configuracao de rotas adicional.

Models

Models de modulos seguem o mesmo padrao do CI3. Como o diretorio do modulo e adicionado ao package_path durante o carregamento, voce pode usar $this->load->model() normalmente de dentro do controller do modulo.

php models/Meu_modulo_model.php 
if (!defined('BASEPATH')) exit('No direct script access allowed');

class Meu_modulo_model extends CI_Model
{
    public function getPorCliente(int $clienteId)
    {
        return $this->db
            ->where('clientes_id', $clienteId)
            ->get('meu_modulo_dados')
            ->row();
    }

    public function salvar(int $clienteId, string $conteudo)
    {
        $existe = $this->db
            ->where('clientes_id', $clienteId)
            ->count_all_results('meu_modulo_dados') > 0;

        if ($existe) {
            return $this->db->update(
                'meu_modulo_dados',
                ['conteudo' => $conteudo],
                ['clientes_id' => $clienteId]
            );
        }
        return $this->db->insert('meu_modulo_dados', [
            'clientes_id' => $clienteId,
            'conteudo'     => $conteudo,
        ]);
    }
}

Views

Views de modulos ficam em modules/<slug>/views/. Como o diretorio do modulo e adicionado ao package_path, o CI3 consegue encontra-las automaticamente:

php controllers/Meu_modulo.php 
$this->data['view'] = 'meu_modulo/index';
return $this->layout();

O CI3 ira procurar por meu_modulo/index.php no diretorio de views do modulo (modules/meu_modulo/views/meu_modulo/index.php) alem do diretorio padrao.

Dentro das views, todas as variaveis passadas pelo controller via $this->data estao disponíveis normalmente, assim como os helpers globais do CI3 (site_url(), base_url(), etc.).

View Hooks

View hooks permitem que um modulo injete conteudo HTML em pontos especificos de views existentes da aplicacao sem modificar os arquivos originais.

Como funcionam

  1. Declarar o hook no modulo

    Crie o arquivo hooks/module_hooks.php no modulo:

    php hooks/module_hooks.php 
    $moduleViewHooks = [
    'os_detail_extras' => [
        FCPATH . 'modules/meu_modulo/views/hooks/os_extras.php',
    ],
];

    O array pode conter multiplos arquivos para o mesmo hook. Todos serao incluidos em ordem.

  2. Chamar o hook na view existente

    Adicione module_hook() nos pontos da view onde o conteudo deve ser injetado. Isso ja foi feito nas views listadas na secao Pontos de Injecao.

    php application/views/os/visualizar.php 
    <!-- ... conteudo existente ... -->
<?php module_hook('os_detail_extras'); ?>
<!-- ... continua ... -->

  3. Escrever o arquivo de hook

    O arquivo incluido roda com acesso a todas as variaveis da view (via extract()) e ao CI via get_instance(). Use sempre $CI = &get_instance() em vez de $this pois o arquivo nao esta dentro de um metodo de classe.

    php views/hooks/os_extras.php 
    $CI  = &get_instance();
$osId = isset($result) ? (int) $result->idOs : (int) $CI->uri->segment(3);
if (!$osId) return;

$dados = $CI->db
    ->where('os_id', $osId)
    ->get('meu_modulo_dados')
    ->row();
?>
<div class="widget-box">
    <div class="widget-title">
        <h5>Dados Extras</h5>
    </div>
    <div class="widget-content">
        <?= $dados ? htmlspecialchars($dados->conteudo) : 'Nenhum dado.' ?>
    </div>
</div>

Pontos de Injecao Disponiveis (Views)

cliente_listagem_th views/clientes/clientes.php — <thead>, antes de "Acoes"
Injetado na linha de cabecalho da tabela de clientes. Use para adicionar um <th>. Nenhuma variavel de contexto; e renderizado uma unica vez.
cliente_listagem_td views/clientes/clientes.php — <tbody>, antes da coluna "Acoes", por linha
Injetado uma vez por linha do foreach de clientes. A variavel $cliente_row contem o objeto da linha atual (injetada via get_instance()->load->vars(['cliente_row' => $r]) antes da chamada). Use para adicionar um <td> por cliente.
cliente_editar_form_extras views/clientes/editarCliente.php — dentro do <form>, antes dos botoes
Injetado dentro do formulario de edicao, imediatamente antes de <div class="form-actions">. Ideal para adicionar campos control-group que se integram visualmente ao formulario existente. A variavel $result contem o objeto do cliente.
cliente_adicionar_form_extras views/clientes/adicionarCliente.php — dentro do <form>, antes dos botoes
Injetado dentro do formulario de criacao de cliente, imediatamente antes de <div class="form-actions">. Campos de input com name definido serao submetidos junto com o formulario. Use o hook de controller clientes_adicionar_after_insert para processar esses campos apos o insert.
cliente_editar_extras views/clientes/editarCliente.php — apos o fechamento do <form>
Injetado apos o fechamento do formulario principal de edicao de cliente. A variavel $result contem o objeto do cliente. Util para widgets independentes ou paineis fora do form.
cliente_visualizar_extras views/clientes/visualizar.php — dentro da aba principal
Injetado dentro da primeira aba da pagina de visualizacao do cliente, apos os acordeons de dados existentes. A variavel $result contem o objeto do cliente. Ideal para adicionar um novo bloco accordion-group.
Adicionando novos pontos de injecao

Para adicionar um novo ponto de injecao em qualquer view existente, insira <?php module_hook('nome_do_hook'); ?> no local desejado. O helper module_hook e carregado automaticamente via autoload. Para passar dados de contexto (ex: objeto de uma linha de loop), use get_instance()->load->vars(['chave' => $valor]) antes da chamada.

Controller Hooks

Controller hooks permitem que um modulo execute logica em pontos especificos de controllers existentes — sem modificar os arquivos originais. Ao contrario dos view hooks (que apenas injetam HTML), os controller hooks podem alterar o fluxo de execucao, adicionar dados a operacoes de save, ou interromper uma operacao inteiramente (ex: bloquear um login).

Ha dois tipos de controller hooks: hooks de proxy (gerados automaticamente para qualquer metodo de qualquer controller) e hooks manuais (inseridos explicitamente no controller para cenarios onde o proxy nao e suficiente).

Sistema de Proxy Automatico

Quando um modulo declara um hook com a convencao before::Controller::metodo ou after::Controller::metodo, o sistema de modulos gera automaticamente uma classe proxy para o controller referenciado. Isso acontece no momento da instalacao e nenhuma alteracao manual no controller e necessaria.

O que o sistema faz ao instalar o modulo

  1. Detecta os controllers necessarios

    O sistema analisa as chaves de $moduleControllerHooks que seguem o padrao before::X::* ou after::X::* e extrai o nome do controller (X).

  2. Faz backup do controller original

    O arquivo application/controllers/Mine.php e copiado para application/controllers/__originals/Mine.php e a classe e renomeada internamente de Mine para Mine__real.

  3. Gera o arquivo proxy

    Um novo application/controllers/Mine.php e gerado. Ele inclui o backup e declara uma classe Mine extends Mine__real onde cada metodo publico e envolvido com as chamadas de hook:

    php application/controllers/Mine.php (gerado) 
    // Auto-gerado pelo Module_manager — nao edite manualmente.
require_once __DIR__ . '/__originals/Mine.php';

class Mine extends Mine__real
{
    public function login()
    {
        $args = func_get_args();
        $ctx  = [
            'controller' => 'Mine',
            'method'     => 'login',
            'args'       => &$args,
            'result'     => null,
        ];

        module_controller_hook('before::Mine::login', $ctx);

        if (!empty($ctx['abort'])) {
            if (isset($ctx['response'])) echo $ctx['response'];
            return;
        }

        $ctx['result'] = parent::login(...$args);
        module_controller_hook('after::Mine::login', $ctx);
        return $ctx['result'];
    }

    // ... todos os outros metodos publicos seguem o mesmo padrao ...
}

  4. Restaura ao desinstalar

    Ao desinstalar o modulo, o sistema verifica se algum outro modulo ativo ainda precisa do proxy. Se nenhum precisar, o backup e restaurado e o controller volta ao estado original.

Limitacao: hooks after:: com redirect()

Quando um metodo do controller original chama redirect(), o PHP executa exit internamente. Isso impede que o hook after:: seja disparado. Para esses casos, use um hook manual inserido explicitamente antes do redirect (ver secao abaixo).

Convencao de nomenclatura

PadraoQuando dispara
before::Controller::metodo Antes de o metodo original ser chamado. Pode setar abort = true para cancelar a execucao.
after::Controller::metodo Apos o metodo original retornar. $context['result'] contem o valor de retorno. Nao dispara se o metodo chamou redirect().

Chaves padrao do $context nos hooks de proxy

ChaveTipoDisponivel emDescricao
controllerstringbefore, afterNome do controller (ex: 'Mine').
methodstringbefore, afterNome do metodo (ex: 'login').
argsarray (ref)before, afterArgumentos passados ao metodo. Modificavel antes do before.
resultmixedafter apenasValor retornado pelo metodo original.
abortboolbefore apenasSetar true cancela a execucao do metodo original.
responsestringbefore apenasConteudo enviado ao cliente quando abort = true.

Hooks Manuais

Para metodos que terminam com redirect() — onde o hook after:: nunca dispararia — o proprio controller declara um ponto de hook explicito antes do redirect. Esses hooks sao convencionados pelo nome do controller e da operacao, sem o prefixo before::.

php application/controllers/Clientes.php 
if ($this->clientes_model->add('clientes', $data) == true) {
    $newId   = $this->db->insert_id();
    $context = ['idClientes' => (int) $newId];
    module_controller_hook('clientes_adicionar_after_insert', $context);
    redirect(site_url('clientes/'));
}

O arquivo de hook do modulo usa as chaves do $context que o controller declara explicitamente:

php hooks/controller/clientes_adicionar.php 
// $context e $CI estao disponiveis aqui
$CI        = &get_instance();
$clienteId = (int) $context['idClientes'];
$permitido = $CI->input->post('login_permitido') !== null ? 1 : 0;

if ($permitido === 0) {
    $CI->db->update('clientes', ['login_permitido' => 0], ['idClientes' => $clienteId]);
}

Convencoes do $context

ChaveTipoUso
abort bool Sete como true para interromper o metodo original (apenas em hooks before::).
response string Conteudo enviado ao cliente quando abort = true. Geralmente JSON.
chaves especificas do hook mixed Dados da operacao. Variam por hook. Documentados na tabela de pontos de hook abaixo.

Declarando controller hooks no modulo

Em hooks/module_hooks.php, declare o array $moduleControllerHooks usando a convencao before::Controller::metodo para hooks de proxy, ou o nome livre para hooks manuais:

php hooks/module_hooks.php 
$moduleControllerHooks = [
    // Hook de proxy — dispara antes de Mine::login()
    'before::Mine::login' => [
        FCPATH . 'modules/meu_modulo/hooks/controller/mine_login.php',
    ],

    // Hook manual — declarado explicitamente em Clientes::adicionar()
    'clientes_adicionar_after_insert' => [
        FCPATH . 'modules/meu_modulo/hooks/controller/clientes_adicionar.php',
    ],
];
Um hook pode ter varios modulos

Varios modulos podem registrar arquivos para o mesmo hook. Eles serao executados em ordem de instalacao. Nenhum modulo precisa saber da existencia dos outros.

Pontos de Hook Disponiveis

Hooks de Proxy (before:: / after::)

O sistema de proxy gera automaticamente hooks before:: e after:: para todos os metodos publicos de um controller quando pelo menos um modulo declara um hook nesse controller. A tabela abaixo lista os mais relevantes.

Hook Controller Quando dispara Notas
before::Mine::login Mine.php Antes de qualquer validacao de credenciais na area do cliente. Use abort = true para bloquear o login antes de verificar a senha.
after::Mine::login Mine.php Apos login() retornar — so dispara se o metodo nao chamou redirect(). Util para post-processamento de login mal-sucedido.
before::Clientes::editar Clientes.php Antes de qualquer logica do metodo editar.
before::Clientes::adicionar Clientes.php Antes de qualquer logica do metodo adicionar.
Como obter hooks para qualquer controller

Qualquer controller pode ser alvo de um hook de proxy. Basta declarar before::NomeDoController::nomeDoMetodo no $moduleControllerHooks do modulo. O sistema gera o proxy automaticamente na instalacao — sem nenhuma modificacao manual.

Hooks Manuais Disponiveis

clientes_adicionar_after_insert Clientes.php :: adicionar() — apos insert bem-sucedido, antes de redirect()

Executado imediatamente apos um novo cliente ser inserido no banco. Nao e um hook de proxy porque adicionar() chama redirect(), o que impediria o after:: de disparar.

Chave em $contextTipoDescricao
idClientesintID do cliente recem-inserido ($this->db->insert_id()).
Adicionando hooks manuais em novos pontos

Para expor um ponto de hook manual em qualquer controller, adicione as linhas abaixo no local desejado. A verificacao de abort so e necessaria se o hook pode interromper a operacao.

php
$context = ['dado_relevante' => $valor];
module_controller_hook('nome_do_hook', $context);
if (!empty($context['abort'])) { echo $context['response']; return; }

Menu Dinamico

Itens de menu declarados no module.json sao inseridos na tabela modulo_menu_items durante a instalacao. O MY_Controller carrega esses itens em cada requisicao autenticada e os disponibiliza como $module_menu_items para o template de menu.

Apenas modulos com status = 'ativo' tem seus itens exibidos. Ao desativar um modulo via toggle, os itens desaparecem do menu imediatamente, sem reinstalacao.

Permissoes de menu

Se o campo permission do item estiver preenchido, o item so aparece para usuarios cujo grupo de permissao inclua aquela permissao. Se estiver vazio, o item e visível para todos os usuarios autenticados.

Configuracoes

Configuracoes declaradas no module.json sao inseridas na tabela configuracoes via INSERT IGNORE, preservando qualquer valor que o usuario tenha alterado manualmente.

Para acessar uma configuracao dentro do modulo:

php 
// Em qualquer controller ou view hook
$valor = $this->data['configuration']['meu_modulo_opcao'];

// Em um arquivo de view hook (via extract)
$valor = $configuration['meu_modulo_opcao'] ?? 'padrao';

API Reference

module_hook(string $hookName)

Executa um ponto de injecao em uma view. Busca os arquivos registrados para $hookName e inclui cada um com as variaveis da view disponíveis no escopo via extract(). Silencioso quando o hook nao tem arquivos registrados.

module_controller_hook(string $hookName, array &$context = [])

Executa um ponto de hook em um controller. Inclui cada arquivo registrado com $CI e $context disponiveis no escopo. O $context e passado por referencia — modificacoes feitas pelo hook file sao refletidas no array original no controller. Silencioso quando o hook nao tem arquivos registrados.

ParametroTipoDescricao
$hookName string Nome do ponto de hook. Ex: 'mine_login_after_verify'.
$context array (ref) Dados do contexto. O hook file pode ler e modificar este array.
ParametroTipoDescricao
$hookName string Nome do ponto de injecao. Ex: 'cliente_editar_extras'.

Module_manager::upload(array $file)

Processa um upload de ZIP. Extrai, valida o module.json e move para modules/<slug>/. Retorna ['slug' => ..., 'manifest' => ...] em caso de sucesso ou ['error' => ...] em caso de falha.

Module_manager::install(string $slug)

Instala um modulo que ja esta em modules/<slug>/. Retorna ['success' => true] ou ['error' => ...].

Module_manager::uninstall(string $slug)

Desinstala o modulo, revertendo todas as alteracoes feitas na instalacao e removendo o diretorio do modulo.

Module_manager::toggle(string $slug)

Alterna o status do modulo entre ativo e inativo. Modulos inativos nao carregam hooks, nao aparecem no menu e seus controllers ainda sao acessiveis mas as permissoes do controller devem bloquear o acesso.

Module_manager::getManifest(string $slug)

Le e decodifica o module.json do modulo localizado em modules/<slug>/module.json. Retorna o manifest como array associativo, ou null se o arquivo nao existir ou for invalido. Usado internamente pelo controller Modulos::visualizar() e Modulos::confirmar() para passar os dados a view.

Controller

php controllers/Cliente_observacao.php 
if (!defined('BASEPATH')) exit('No direct script access allowed');

class Cliente_observacao extends MY_Controller
{
    public function __construct()
    {
        parent::__construct();

        if (!$this->permission->checkPermission(
            $this->session->userdata('permissao'),
            'eCliente'
        )) {
            $this->output->set_status_header(403);
            exit(json_encode(['error' => 'Sem permissao.']));
        }
    }

    /** GET /cliente_observacao/get/<id> */
    public function get(int $clienteId)
    {
        $row = $this->db
            ->where('clientes_id', (int) $clienteId)
            ->get('clientes_observacao')
            ->row();

        $this->output
            ->set_content_type('application/json')
            ->set_output(json_encode([
                'observacao' => $row ? $row->observacao : ''
            ]));
    }

    /** POST /cliente_observacao/salvar */
    public function salvar()
    {
        $clienteId  = (int)    $this->input->post('clientes_id');
        $observacao = (string) $this->input->post('observacao');

        if (!$clienteId) {
            $this->output
                ->set_content_type('application/json')
                ->set_output(json_encode(['error' => 'ID invalido.']));
            return;
        }

        $existe = $this->db
            ->where('clientes_id', $clienteId)
            ->count_all_results('clientes_observacao') > 0;

        if ($existe) {
            $this->db->update(
                'clientes_observacao',
                ['observacao' => $observacao],
                ['clientes_id' => $clienteId]
            );
        } else {
            $this->db->insert('clientes_observacao', [
                'clientes_id' => $clienteId,
                'observacao'  => $observacao,
            ]);
        }

        $this->output
            ->set_content_type('application/json')
            ->set_output(json_encode(['success' => true]));
    }
}

Troubleshooting

Controller do modulo retorna 404

Verifique se o arquivo wrapper foi criado em application/controllers/<Slug>.php. Caso o diretorio nao tenha permissao de escrita para o servidor web, o sistema de fallback via $route['404_override'] = 'module_dispatch' assume o roteamento. Confirme que Module_dispatch.php existe em application/controllers/.

View hook nao aparece na pagina

  1. Confirme que o modulo esta com status = 'ativo' no banco.
  2. Confirme que hooks/module_hooks.php existe e retorna o array $moduleViewHooks correto.
  3. Confirme que o arquivo de hook referenciado existe em disco.
  4. Confirme que a view existente chama <?php module_hook('nome'); ?> no local correto.
  5. No arquivo de hook, use sempre $CI = &get_instance() em vez de $this.
  6. Para acessar o ID do recurso, use o fallback de URI: $id = isset($result->id) ? $result->id : (int) $CI->uri->segment(3);

SQL de instalacao falha silenciosamente

O Module_manager retorna ['error' => ...] com a mensagem do banco em caso de falha. Verifique os logs do CI3 em application/logs/ e certifique-se de que o usuario do banco tem permissao CREATE TABLE.

Upload do ZIP falha com "No such file or directory"

Isso ocorre quando o ZIP contem uma subpasta raiz e a logica de unwrap falha. Certifique-se de que o diretorio modules/ tem permissao de escrita para o usuario do servidor web (www-data em sistemas Debian/Ubuntu).

Modulo desinstalado mas tabelas ainda existem

Verifique se migrations/uninstall.sql contem os comandos DROP TABLE IF EXISTS para todas as tabelas criadas pelo modulo. O arquivo e obrigatorio mesmo que esteja em branco; sua ausencia nao gera erro, mas as tabelas nao serao removidas.

Permissoes de menu nao funcionam

O campo permission do item de menu deve corresponder exatamente a uma permissao cadastrada no sistema de grupos. Permissoes customizadas de modulos precisam ser inseridas manualmente via install.php na tabela de permissoes do grupo do administrador.
